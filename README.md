# devsys-14-devops-intro
## Описание истории
Представьте, что вы работаете в стартапе, который запустил интернет-магазин. Ваш интернет-магазин достаточно успешно развивался, и вот пришло время налаживать процессы: у вас стало больше конечных клиентов, менеджеров и разработчиков.Сейчас от клиентов вам приходят задачи, связанные с разработкой нового функционала. Задач много, и все они требуют выкладки на тестовые среды, одобрения тестировщика, проверки менеджером перед показом клиенту. В случае необходимости, вам будет необходим откат изменений.

## Решение задачи
1. Менеджер (либо аналитик\лид разработки) декомпозирует задачи
2. На очередном планировании спринта команда приоретизирует задачи и берет их в работу
3. Идет процесс разработки нового функционала. В рамках разработки необходимо покрытие кода unit тестами.
4. QA-инженеры реализуют авто-тесты и интеграционные тесты для нового функционала.
5. DevOps-инженеры развертывают среды test, stage. Среды максимально схожи с текущим продуктивом (в идеале являются клоном продуктива)
6. DevOps-инженеры реализуют CI/CD пайплайн, включающий запуск unit тестов и авто-тестов, написанных QA-инженерами, проверку кода статическим анализатором и SAST инструментами. Код не прошедший проверки, либо упавший на тестах в рамках CI процесса, должен дорабатываться. DevOps-инженеры реализуют пайплайн доставки нового функционала на среды test, stage, prod с возможностью восстановления.
7. По мере закрытия задач, в рамках Continuous Delivery артефакты попадают на test среды.
8. QA-инженеры выполняют regress-тестирование на test среде.
9. При отсутствии проблем на этапе тестирования новый функционал деплоится на stage.
10. Проводится интеграционное тестирование и приемные испытания с проверкой менеджером.
11. При отсутствии проблем на предыдущем этапе новый функционал деплоится на prod.
